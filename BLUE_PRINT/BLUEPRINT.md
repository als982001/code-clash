# Code Clash - 프로젝트 블루프린트

## 1. 기술 스택

| 영역                 | 기술                                   | 비고                              |
| -------------------- | -------------------------------------- | --------------------------------- |
| Framework            | Next.js                                |                                   |
| Styling              | Tailwind CSS + shadcn/ui               |                                   |
| State Management     | Tanstack Query (v5+) + Zustand         |                                   |
| Database + Real-time | Supabase                               |                                   |
| Auth                 | Supabase Auth                          |                                   |
| Editor               | Monaco Editor (`@monaco-editor/react`) |                                   |
| Code Sandbox         | Judge0 API                             |                                   |
| AI SDK               | Vercel AI SDK + Claude API or Google   | 코드 리뷰, 다음 단계 문제 추천 등 |
| 커뮤니케이션         | Socket.io                              |                                   |

## 2. 추가 설계 필요 항목

- 데이터 스키마
- 핵심 로직 및 정책
- 시스템 아키텍처 및 통신 프로토콜

- 디자인 컨셉 및 게임 UI → Stitch 이용 예정

## 3. 필요한 화면 및 경로

### 3.1 랜딩 및 온보딩 (`/`)

- 서비스 소개 및 실시간 경쟁 현황 요약 노출
- 소셜 로그인을 통한 진입
- 비로그인 유저를 위한 연습 모드 → **MVP 아님**

### 3.2 메인 대시보드/로비 (`/dashboard`)

- **유저 요약**: 현재 티어, MMR, 최근 승률, 연속 풀이 일수 표시 등
- **문제 리스트**: 난이도별, 태그별 알고리즘 문제 목록 필터링 및 검색
- **매칭 시작**: 실시간 대전 찾기 버튼 클릭 시 매칭 큐 진입
- **랭킹 요약**: 상위 랭커들 리스트 노출

### 3.3 실시간 대전 방/에디터 (`/play/[matchId]`)

- **코드 에디팅**: Monaco Editor를 통한 실시간 코드 작성 및 자동 완성
- **문제 지문**: 좌측 또는 우측 섹션에 문제 설명 및 제약 사항 표시
- **실시간 진행률**: 상대방이 몇 번째 테스트 케이스를 통과 중인지 프로그레스 바 등으로 실시간 시각화
- **채점 및 제출**: '코드 실행'을 통한 테스트 케이스 확인 및 '최종 제출' 시 상대방과 점수 비교
- **타이머**: 남은 시간을 실시간으로 카운트다운

### 3.4 결과 및 AI 리뷰 (`/result/[matchId]`)

- **승패 판정**: 소요 시간, 통과 여부, 효율성을 종합한 최종 스코어 공개
- **MMR 변동**: 결과에 따른 랭킹 점수 업데이트 연출
- **AI 상세 리뷰 (Gemini)**: 제출한 코드를 분석하여 시간 복잡도 개선 방안, 가독성 피드백, 더 나은 접근 방식 제안
- **상대방 코드 보기**: (옵션) 경쟁이 끝난 후 서로의 코드를 공유하며 학습

### 3.5 리더보드 (`/leaderboard`)

- 전체 유저 중 자신의 위치 확인하는 커뮤니티 성격
- 전체/시즌별 랭킹 순위 표시
- 친구 추가 혹은 특정 유저의 프로필로 이동 가능

### 3.6 개인 프로필 및 통계 (`/profile/[userId]`)

- **풀이 히스토리**: 그동안 참여한 대전 기록 및 성공/실패 여부
- **역량 분석**: 그래프(방사형 차트 등)를 통해 알고리즘 유형별(그리디, DP 등) 강점/약점 시각화
- **북마크**: 나중에 다시 풀어보고 싶은 문제 저장 목록

## 4. 설계 시 고려할 인터랙션 포인트

- **대전 방 진입 전**: 매칭 중에는 유저가 지루하지 않도록 '오늘의 팁'이나 '간단한 퀴즈'를 띄워주는 인터랙션 필요
- **에디터 화면**: 폰트 크기 조절, 다크/라이트 모드 전환, 코드 초기화 등 사용자 편의 기능 포함
- **실시간 피드백**: 소켓 통신을 통해 "상대방이 코드를 제출했습니다!"와 같은 알림을 상단에 띄워 긴장감 유지

## 5. 핵심 로직 및 정책

### 5.1 점수 산출 공식

완성도(통과한 테스트 케이스 비율)를 최우선으로 하되, 남은 시간을 가산점으로 부여하는 방식입니다.

```
Score = (C_passed / C_total × 1000) + ((T_max - T_used) / T_max × 500)
```

| 항목                                         | 배점         | 설명                                                                                      |
| -------------------------------------------- | ------------ | ----------------------------------------------------------------------------------------- |
| **완성도** (`C_passed / C_total`)            | 최대 1,000점 | 전체 테스트 케이스 중 통과한 비율. 코드의 정확성을 가장 높게 평가                         |
| **시간 보너스** (`(T_max - T_used) / T_max`) | 최대 500점   | 전체 제한 시간 대비 남은 시간의 비율. 같은 문제를 풀더라도 더 빨리 푼 사람이 높은 점수    |
| **효율성 가중치** (선택)                     | +1점         | 동점 시 코드의 실행 시간(Runtime)이나 메모리 사용량이 적은 쪽에 추가 부여하여 변별력 확보 |

### 5.2 실시간 매칭 정책

- **Elo 레이팅 기반 그룹화**: 유저의 실력을 점수화(MMR)하여 관리

- **단계별 검색 범위 확대**:
  | 시간 | 검색 범위 |
  |------|-----------|
  | 0~10초 | 내 MMR 기준 ±50점 |
  | 11~20초 | ±150점으로 확대 |
  | 21~30초 | 전체 범위 확대 |

- **AI 대전 전환** → 고려 후 결정

### 5.3 승패 판정 및 예외 처리

| 상황          | 판정 기준                                                                           |
| ------------- | ----------------------------------------------------------------------------------- |
| 정상 종료     | 점수 높은 유저 승리                                                                 |
| 동점          | 먼저 최종 제출 버튼 누른 유저 승리 (`submitted_at` 타임스탬프 비교)                 |
| 중도 퇴장     | 퇴장한 유저 즉시 패배 처리                                                          |
| 네트워크 단절 | 30초 재접속 유예 기간 부여 후 미접속 시 패배                                        |
| 시간 초과     | 제출 시점까지의 테스트 케이스 점수로 계산. 둘 다 못 풀었을 경우 완성도 높은 쪽 승리 |

### 5.4 채점 파이프라인

프론트엔드에서 제출된 코드가 결과로 돌아오기까지의 이벤트 기반 흐름:

1. **Submit (FE)**: 유저가 작성한 코드와 언어 정보를 백엔드(Next.js API Route)로 전송
2. **Queueing (BE)**: 채점 요청을 큐에 적재 (유저가 몰릴 경우 서버 과부하 방지)
3. **Execution (Sandbox)**: Judge0와 같은 외부 채점 엔진에 코드 전달
   - Input: 소스코드, 표준 입력(테스트 케이스), 시간/메모리 제한
   - Output: 통과 여부, 실행 시간, 메모리 사용량, 에러 메시지(Runtime/Compile)
4. **Score Calc**: 반환된 결과값을 바탕으로 서버에서 최종 점수 계산
5. **Broadcast (Real-time)**: 계산된 점수와 승패 결과를 Supabase Realtime이나 Socket.io를 통해 양쪽 유저에게 동시 전송
6. **Review Mode**: 결과 화면으로 전환되며, 동시에 Gemini API가 백그라운드에서 코드 리뷰 생성 시작

## 6. 시스템 및 아키텍처 통신

> 핵심 원칙: **"모든 실시간 상태의 원천(Source of Truth)은 DB가 아닌 메모리(또는 서버)에 있고, 클라이언트는 이를 반영하기만 한다"**

### 6-1. 시스템 아키텍처 (System Architecture)

전체적인 구조는 Next.js를 중심으로 하되, 실시간 데이터 동기화는 Supabase Realtime이 담당하고, 무거운 채점 로직은 외부 엔진(Judge0)이 처리하는 구조입니다.

| 레이어           | 기술               | 역할                                                                                |
| ---------------- | ------------------ | ----------------------------------------------------------------------------------- |
| Frontend         | Next.js            | UI 렌더링, Monaco Editor 관리, 클라이언트 사이드 타이머, 실시간 이벤트 수신 및 반영 |
| Backend          | Next.js API Routes | 보안이 필요한 로직 처리 (점수 계산, Judge0 API 호출, Gemini API 호출)               |
| Real-time Layer  | Supabase Realtime  | 대전 중인 두 유저 간의 상태(상대방 진행률, 매칭 성공 알림 등)를 0.1초 단위로 동기화 |
| Database         | PostgreSQL         | 유저 정보, 문제 은행, 대전 기록(Match History) 저장                                 |
| Execution Engine | Judge0             | 격리된 환경에서 유저 코드를 실행하고 시간/메모리 사용량 반환                        |
| AI Engine        | Google AI Studio   | 대전 종료 후 코드 리뷰 및 다음 문제 추천                                            |

### 6-2. 통신 프로토콜: 실시간 이벤트 목록

실시간 대전은 단순히 데이터를 주고받는 것을 넘어, **이벤트 기반(Event-driven)**으로 동작해야 합니다. Supabase Realtime의 Broadcast 기능을 활용할 때 사용할 주요 이벤트 명세입니다.

#### A. 매칭 및 준비 (Matchmaking)

| 이벤트          | 설명                                                      |
| --------------- | --------------------------------------------------------- |
| `MATCH_REQUEST` | 유저가 대기열에 진입함                                    |
| `MATCH_FOUND`   | 서버가 대전 상대를 찾았음을 양측에 알림 (대전 방 ID 전달) |
| `PLAYER_READY`  | 두 유저가 문제 로딩을 마치고 시작 버튼을 눌렀을 때 동기화 |

#### B. 인게임 인터랙션 (In-Game)

| 이벤트               | 설명                                                                         | Payload                               |
| -------------------- | ---------------------------------------------------------------------------- | ------------------------------------- |
| `PROGRESS_UPDATE`    | 상대방이 테스트 케이스를 몇 개 통과했는지 실시간 전송 (프로그레스 바 갱신용) | `{ userId, passedCount, totalCount }` |
| `OPPONENT_SUBMITTED` | 상대방이 최종 제출을 완료했다는 신호 (긴장감 조성용)                         |                                       |
| `GAME_TICK`          | (필요 시) 서버-클라이언트 간 타이머 오차 보정을 위한 동기화 신호             |                                       |

#### C. 결과 및 종료 (Game End)

| 이벤트           | 설명                                                                 | Payload                                                                   |
| ---------------- | -------------------------------------------------------------------- | ------------------------------------------------------------------------- |
| `MATCH_FINISHED` | 양쪽 모두 제출했거나 제한 시간이 종료되었을 때 서버가 승패 결과 전송 | `{ winnerId, scores: { user1: score, user2: score }, mmrChange: number }` |

### 6-3. 상세 데이터 흐름 (Judging Flow)

유저가 '제출' 버튼을 눌렀을 때 일어나는 백엔드 시퀀스입니다.

1. **FE → BE**: 유저의 소스코드와 `matchId`를 POST 요청으로 전송
2. **BE → Judge0**: API를 통해 코드를 전달하고 결과를 기다림 (Sync/Async 선택 가능)
3. **Judge0 → BE**: 각 테스트 케이스별 통과 여부, 실행 시간, 메모리 결과 반환
4. **BE (Logic)**: 앞서 정의한 점수 산출 공식에 따라 최종 점수 계산
5. **BE → DB**: 해당 매칭의 결과와 제출 이력을 저장
6. **BE → Realtime Channel**: 계산된 점수와 결과 페이지 이동 신호를 대전 방의 모든 유저에게 Broadcast

### 6-4. 보안 및 예외 처리 규약

- **Anti-Cheat**: 모든 점수 계산과 승패 판정은 반드시 **Server-side (API Routes)**에서 수행. 클라이언트는 결과만 표시
- **Idempotency (멱등성)**: 네트워크 문제로 유저가 제출 버튼을 여러 번 눌러도, 동일한 `matchId`와 `userId`에 대해서는 첫 번째 제출만 유효하도록 처리
- **Rate Limiting**: 유저가 무분별하게 '코드 실행'을 눌러 Judge0 API 비용이 폭증하는 것을 방지하기 위해, 실행 간격(예: 3초) 제한

## 7. AI 활용 전략

> 핵심 원칙: **"AI는 도구일 뿐, 유저의 '문제 풀이 경험'을 방해해서는 안 된다"**

### 7-1. AI 역할 정의

프로젝트 내에서 AI는 단순한 챗봇이 아닌, 콘텐츠 생성기, 코칭 전문가, 가상 플레이어의 3가지 핵심 역할을 수행합니다.

- 콘텐츠 생성 외의 모델은 **프로젝트 여건 상 추가되지 않을 수도 있습니다.**
- 하단의 활용 모델은 **프로젝트 여건 상 변경될 수 있습니다.**

| 역할            | 활용 모델      | 설명                                                        |
| --------------- | -------------- | ----------------------------------------------------------- |
| 콘텐츠 생성     | Gemini 3 Pro   | 유저 수준에 맞는 고품질 알고리즘 문제 및 테스트 케이스 생성 |
| 심층 분석       | Gemini 3 Pro   | 대전 종료 후 코드 효율성 분석 및 개선안 제안                |
| 실시간 인터렉션 | Gemini 3 Flash | 매칭 대기 중 AI 대전 상대(Bot) 및 게임 중 힌트 제공         |

### 7-2. 핵심 AI 기능 (Core AI Features)

#### A. 다이내믹 문제 생성기 (AI Problem Generator)

정적인 문제 은행의 한계를 극복하기 위해, AI가 실시간으로 문제를 생성하거나 기존 문제를 변형합니다.

- **작동 방식**: 특정 주제(예: DFS/BFS)와 난이도(Silver ~ Gold)를 입력하면 Gemini가 JSON 포맷으로 문제 지문, 제약 조건, 입출력 예시를 출력
- **프롬프트 전략**: "너는 알고리즘 출제 위원이야. 특정 시간 복잡도($O(N \log N)$)를 강제하는 효율성 테스트 케이스를 포함해줘"라는 System Instruction 활용

#### B. 지능형 포스트-매치 리뷰 (Post-Match Analysis)

경쟁이 끝난 후 유저가 작성한 코드를 분석하여 성취감을 극대화합니다.

- **코드 비교 분석**: "상대방의 코드는 이 부분이 더 효율적이었어" 또는 "너의 코드는 가독성 면에서 훌륭해"와 같은 피드백 제공
- **시간/공간 복잡도 가시화**: 텍스트 설명뿐만 아니라 유저 코드의 복잡도를 수치화하여 데이터로 반환
- **Next Step 추천**: 틀린 원인을 분석하여 "너는 현재 '그리디' 알고리즘에 약점이 있으니, 다음엔 이런 문제를 풀어봐"라고 가이드

#### C. AI 쉐도우 복싱 (AI Opponent Mode)

매칭 인원이 부족할 때 유저의 이탈을 막기 위한 기능입니다.

- **휴먼 시뮬레이션**: Gemini가 해당 문제의 정답 코드를 단계적으로 작성하는 척하며 유저와 경쟁
- **난이도 조절**: 유저의 과거 데이터를 기반으로 AI의 타이핑 속도나 오답 확률을 조절하여 실제 사람과 대결하는 듯한 긴장감 유지

### 7-3. 프롬프트 설계 가이드 (System Prompting)

구글 AI Studio에서 설정할 System Instruction의 핵심 구조입니다.

> **[System Instruction 예시]**
>
> - **Persona**: 10년 차 시니어 풀스택 개발자이자 알고리즘 코치
> - **Objective**: 유저의 코드를 직접 수정해주지 말고, 논리적 허점을 짚어주는 힌트만 제공할 것
> - **Output Format**: 모든 응답은 프론트엔드에서 파싱하기 쉽도록 엄격한 JSON 형식을 유지할 것
> - **Constraint**: 한국어 응답을 원칙으로 하되, 기술 용어는 영어와 병기할 것

### 7-4. AI 연동 아키텍처 및 비용 최적화

- **JSON Mode 활용**: Gemini 3의 구조화 출력 기능을 활용하여 프론트엔드에서 별도의 파싱 로직 없이 바로 UI에 렌더링할 수 있게 설계
- **캐싱 전략 (Caching)**: 동일한 난이도와 주제의 문제는 AI에게 매번 묻지 않고, 한 번 생성된 문제를 Supabase에 저장하여 다른 유저들에게 재사용함으로써 API 호출 비용 절감
- **토큰 관리**: 코드 리뷰 시에는 전체 코드가 아닌 핵심 로직 부분만 추출하여 보냄으로써 토큰 소모 최적화

## 8. 데이터 스키마 설계

> 모든 테이블은 기본적으로 `id (uuid, PK)`와 `created_at (timestamptz)` 필드를 포함합니다.

### 8-1. 유저 및 프로필 (`profiles`)

Supabase Auth의 `auth.users` 테이블과 1:1로 대응되는 확장 프로필 테이블입니다.

| 필드명       | 타입      | 설명                                      |
| ------------ | --------- | ----------------------------------------- |
| `id`         | uuid (PK) | `auth.users.id`와 외래키 연결             |
| `nickname`   | text      | 서비스 내 노출될 이름 (unique)            |
| `avatar_url` | text      | 프로필 이미지 경로                        |
| `mmr`        | integer   | 유저의 실력 점수 (기본값 1000)            |
| `tier`       | text      | MMR에 따른 등급 (Bronze, Silver, Gold 등) |
| `wins`       | integer   | 총 승리 횟수                              |
| `losses`     | integer   | 총 패배 횟수                              |
| `streak`     | integer   | 현재 연속 풀이(승리) 횟수                 |
| `bio`        | text      | 한 줄 소개                                |

### 8-2. 문제 은행 (`problems`)

AI가 생성하거나 미리 등록된 알고리즘 문제 데이터입니다.

| 필드명            | 타입      | 설명                           |
| ----------------- | --------- | ------------------------------ |
| `id`              | uuid (PK) |                                |
| `title`           | text      | 문제 제목                      |
| `description`     | text      | 문제 본문 (Markdown)           |
| `difficulty`      | text      | 난이도 (Level 1, 2, 3 등)      |
| `time_limit`      | integer   | 실행 시간 제한 (ms)            |
| `memory_limit`    | integer   | 메모리 사용 제한 (KB)          |
| `tags`            | text[]    | 문제 유형 태그 (DP, Greedy 등) |
| `is_ai_generated` | boolean   | AI 생성 여부                   |

### 8-3. 테스트 케이스 (`test_cases`)

채점을 위한 입력값과 기대값입니다.

| 필드명            | 타입      | 설명                               |
| ----------------- | --------- | ---------------------------------- |
| `id`              | uuid (PK) |                                    |
| `problem_id`      | uuid (FK) | `problems.id` 참조                 |
| `input`           | text      | 표준 입력값                        |
| `expected_output` | text      | 정답 출력값                        |
| `is_hidden`       | boolean   | 유저에게 숨겨진 테스트 케이스 여부 |

### 8-4. 대전 기록 (`matches`)

실시간 대전의 상태와 결과를 저장합니다.

| 필드명       | 타입        | 설명                                   |
| ------------ | ----------- | -------------------------------------- |
| `id`         | uuid (PK)   |                                        |
| `status`     | text        | 대전 상태 (waiting, ongoing, finished) |
| `winner_id`  | uuid (FK)   | 승리한 유저의 `profiles.id`            |
| `problem_id` | uuid (FK)   | 대전에 사용된 `problems.id`            |
| `start_time` | timestamptz | 실제 게임 시작 시각                    |
| `end_time`   | timestamptz | 게임 종료(또는 제한 시간 종료) 시각    |

### 8-5. 대전 참가자 정보 (`match_participants`)

특정 대전에 참여한 유저들의 개별 성과를 기록합니다. (1:N 관계)

| 필드명            | 타입      | 설명                                          |
| ----------------- | --------- | --------------------------------------------- |
| `id`              | uuid (PK) |                                               |
| `match_id`        | uuid (FK) | `matches.id` 참조                             |
| `user_id`         | uuid (FK) | `profiles.id` 참조                            |
| `score`           | float     | 블루프린트에 정의된 공식으로 산출된 최종 점수 |
| `mmr_change`      | integer   | 해당 판 결과로 변동된 MMR 수치                |
| `is_disconnected` | boolean   | 중도 퇴장 여부                                |

### 8-6. 제출 이력 (`submissions`)

유저가 제출한 코드와 채점 엔진(Judge0)의 결과입니다.

| 필드명         | 타입        | 설명                                       |
| -------------- | ----------- | ------------------------------------------ |
| `id`           | uuid (PK)   |                                            |
| `user_id`      | uuid (FK)   | `profiles.id` 참조                         |
| `match_id`     | uuid (FK)   | `matches.id` 참조                          |
| `code`         | text        | 제출한 소스코드 원문                       |
| `language`     | text        | 사용 언어 (javascript, python 등)          |
| `status`       | text        | 채점 결과 (Accepted, Wrong Answer, TLE 등) |
| `runtime`      | float       | 실행 시간 (ms)                             |
| `memory`       | float       | 메모리 사용량 (KB)                         |
| `passed_cases` | integer     | 통과한 테스트 케이스 개수                  |
| `total_cases`  | integer     | 전체 테스트 케이스 개수                    |
| `submitted_at` | timestamptz | 제출 시각 (동점 판정 시 활용)              |

### 8-7. AI 리뷰 데이터 (`ai_reviews`)

Gemini가 분석한 코드 리뷰 결과입니다.

| 필드명          | 타입      | 설명                                               |
| --------------- | --------- | -------------------------------------------------- |
| `id`            | uuid (PK) |                                                    |
| `submission_id` | uuid (FK) | `submissions.id` 참조 (1:1)                        |
| `content`       | jsonb     | 리뷰 본문, 효율성 분석, 개선 제안 등 구조화 데이터 |
| `summary`       | text      | 결과 화면에 노출할 짧은 요약                       |

### 8-8. 설계 핵심 포인트

- **JSONB 활용**: `ai_reviews.content`를 jsonb 타입으로 설정하여, Gemini가 생성하는 다양한 형태의 분석 데이터를 유연하게 저장
- **타임스탬프 정밀도**: `submissions.submitted_at`은 **동점 판정 로직(먼저 제출한 사람 승리)**을 구현하기 위해 필수적인 데이터
- **데이터 무결성**: `match_participants`와 `submissions`를 분리하여, 한 대전 내에서 여러 번의 '코드 실행'을 거쳐 최종 '제출'을 하는 과정을 모두 기록·관리하기 용이하게 구성
- **역량 분석 지원**: `profiles`와 `submissions`의 관계를 통해 특정 유저가 어떤 태그(`problems.tags`)의 문제를 많이 맞혔는지 쿼리하여 **역량 분석 그래프(방사형 차트)** 구현 가능
