## DB / RLS 상태 (요약)

> 상세 검증은 `CLAUDE.md` "Supabase DB 상태 검증 규칙" 참고.
> 테이블별 row/정책 카운트는 `PROJECT_STATUS.md` "DB 상태" 섹션 참고.

- 테이블: `profiles`, `matches`, `match_participants`, `submissions`, `problems`, `test_cases`, `ai_reviews` 모두 RLS enabled
- `profiles` RLS: `profiles_authenticated_read` (TO authenticated USING true — anon 차단, PR #14) + `self_update` + `self_insert` (PR #7-A 보강)
- `problems` RLS: `public_read` (PR #8)
- `test_cases` RLS: `visible_read` (PR #8 — 히든은 service role 전용)
- `ai_reviews` RLS: `self_read` (SELECT, PR #8 — `submission_id IN (...)` + `TO authenticated`). **write 정책 부재 = 인가 사용자 INSERT default deny → 리뷰 저장은 service-role 단독 경로(write primitive 방지, PR #20). `submission_id` UNIQUE(`ai_reviews_submission_id_key`) 기존재**
- `matches` RLS: `matches_self_or_participant_read` (TO authenticated, `host_id = auth.uid()` OR participant — anon 차단, PR #14) / `anon_insert` / `participant_delete_waiting`. **participant_update 정책은 PR #16 1차 도입 후 winner write primitive 결함으로 같은 PR 후속 커밋에서 DROP — 인가 사용자 UPDATE 는 default deny, matches finalize 는 service-role 전용**
- `match_participants` RLS: `match_participants_co_participant_read` (TO authenticated, 자기 row OR 같은 매치 참가자 row — anon 차단, PR #14) / `self_insert` / `self_delete`. **self_update 정책은 PR #16 1차 도입 후 score write primitive 결함으로 같은 PR 후속 커밋에서 DROP — 인가 사용자 UPDATE 는 default deny, score 갱신은 service-role 전용**
- **BEFORE UPDATE 트리거** (PR #16): `prevent_protected_matches_update` (matches 7개 컬럼 OLD 고정) / `prevent_protected_match_participants_update` (match_participants 6개 컬럼 OLD 고정). **두 테이블 모두 winner/score write primitive fix 후 인가 사용자 UPDATE 정책이 부재하므로 인가 사용자 흐름은 트리거 미도달**, service-role 우회 안전망 역할. 양쪽 모두 `auth.role() = 'service_role'` 분기로 service-role 호출 우회
- **RPC** (2개):
  - `get_invite_match_by_token(p_token text)` — SECURITY DEFINER STABLE, anon+authenticated EXECUTE. `/invite/[token]` 비인증 페이지가 RLS 좁힘 후에도 토큰 검증 가능. 반환 컬럼에서 `invite_token` 제외 (PR #14)
  - `get_profile_stats(p_user_id uuid)` — SECURITY DEFINER STABLE, anon+authenticated EXECUTE. matches/match_participants RLS가 본인 외 데이터 SELECT 차단하므로 타인 프로필 진입 시 전적 0/0/0 표시 회귀를 막기 위해 도입. 반환: `{wins, losses, draws, total_finished}` (PR #18)
- `auth.users` AFTER INSERT 트리거: `handle_new_user` 적용 완료
- `matches`: `invite_token` UNIQUE + `invite_expires_at` + `host_id` 컬럼 (PR #6 선반영, PR #7-D/#14에서 사용)
- `supabase_realtime` publication: `matches` + `match_participants` 둘 다 포함 (PR #7-D `useMatchStatus` postgres_changes 구독에 사용)
- 시드: `problems` 9건 + `test_cases` 43건 (visible 27 / hidden 16) — 멱등 마이그레이션 SoT 확보 (PR #8)
